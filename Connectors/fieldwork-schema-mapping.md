# FieldWise — Booklet → Schema Mapping

How the 2025 Year 10 Phillip Island booklet maps onto the data model, and how the
same data drives the **Plan → Fill → Report** flow. Backend: PostgreSQL (Supabase) + Prisma.

---

## The one idea that makes this work

The booklet, the student's answers, and the final report are **not three documents** —
they are three views over one object:

- **Template** = the blueprint the teacher plans (replaces the current Plan page).
- **FieldworkInstance** = one student's filled-in copy (the Fill stage).
- **Report** = a rendered *view* over that instance's responses (the Report stage) → uploaded to school.

Because responses are structured data, "finish report → upload to school" is one action,
not a re-typing job.

---

## Block-type taxonomy (derived from the booklet)

The 50 pages reduce to **10 reusable block types**. Every site repeats the same pattern,
so once these exist, the whole booklet is just configuration.

| Block type | Where it appears in the booklet | `config` holds | `value` (student answer) |
|---|---|---|---|
| `INSTRUCTION` | Cover, source images, hyperlinks, "STOP" intros | image ref, source URL | none |
| `SHORT_TEXT` | Location, date/time, weather | placeholder | string |
| `LONG_TEXT` | "Describe…", "Evaluate…", ecotourism management Qs | min length | string |
| `TABLE` | Natural/Human features; Management/Purpose/Working | column defs, minRows | array of row objects |
| `RATING_SCALE` | 1–5 ranking (human impact, litter, vegetation, mgmt) | row labels, 1–5 range | { rowId: 1–5 } |
| `CHECKLIST` | NEAP accreditation criteria (8 items) | the 8 criteria | { itemId: bool } |
| `SKETCH_MAP` | "Draw an annotated sketch map… BOLTSSNA" | conventions hint | media ref (canvas export) |
| `FIELD_DATA` | Site 5 coastal measurements | field defs + units | { fieldId: {val, unit} } |
| `PHOTO` | Time-lapse / site photos | gps required? | media refs |
| `GATE` | "STOP: completed all tasks for Site X?" | — | { confirmed: bool } |

## Page-by-page mapping

### Part A — Background (pages 1–6) → `Section` (siteId = null)
- p1 Cover → Template fields (title, subtitle, scale, emergencyContact, copyright)
- p2 Location of island / 5 sites → `LONG_TEXT` ×2 + source image `INSTRUCTION`
- p3 Landscape description + bird's-eye map → `LONG_TEXT` + `SKETCH_MAP`
- p4 Indigenous history + ecotourism concept → `LONG_TEXT` ×2
- p5 Tourism economy → `LONG_TEXT`
- p6 Eco-certification / NEAP → `INSTRUCTION` (table image) + `LONG_TEXT` ×2

### Part B — the repeating **Site pattern** (Sites 1–5) → one `Section` per `StudySite`

Standard per-site block sequence (Sites 1–4):
1. `INSTRUCTION` — site intro + source image
2. `TABLE` — Natural Features | Human Features
3. `TABLE` — Management Technique | Purpose | Is it working?  (`minRows` = 3 or 5)
4. `RATING_SCALE` — 4 rows, 1–5
5. `LONG_TEXT` ×3–5 — resource/people management, educational purpose
6. `CHECKLIST` — 8 NEAP criteria
7. `LONG_TEXT` — accreditation rating + why
8. `SKETCH_MAP` — annotated, BOLTSSNA
9. `LONG_TEXT` ×2–3 — ecotourism management + education measures + evaluation
10. `GATE` — "STOP: completed all tasks for Site X?"

Sites and coordinates (fill in exact lat/long during setup):
| order | StudySite.name | pages |
|---|---|---|
| 1 | Koala Conservation Centre | 7–14 |
| 2 | San Remo Pelican Feeding | 15–21 |
| 3 | Churchill Island | 22–29 |
| 4 | The Nobbies & Antarctic Journey | 30–37 |
| 5 | Summerland Bay & Penguin Parade Visitor Centre | 38–46 |

> Note: the source PDF has some mislabelled "Site X" cross-references (e.g. p7 says
> "Site 4", p30 "Site 3 4"). Structuring sites as real objects removes this class of
> error — the app numbers them, not the author.

**Site 5 is the exception** — it adds a `FIELD_DATA` block for coastal measurements
(beach width, sand description, wind/wave direction, wave height/length/period, swash
distance, constructive vs destructive) plus a `PHOTO` block hinting at time-lapse capture.

### Part C — Synthesis (pages 48–50) → `Section` (siteId = null)
- p48 Future management suggestions → `LONG_TEXT`
- p49–50 Notes → `LONG_TEXT` (freeform)

## How each stage uses the model

**Plan (teacher).** Teacher creates a `Template`, adds `StudySite`s (name + GPS),
and arranges `QuestionBlock`s. Cloning next year = copy the Template, bump `version`.
This is what replaces the Plan page.

**Fill (student).** Teacher issues an `Assignment` to a `ClassGroup`; each student gets a
`FieldworkInstance`. Every answer is a `Response` (typed `value` JSON) linked to its block.
Photos/sketches become `MediaAsset`s with GPS + timestamp. `GATE` blocks drive the
"have you finished this site?" completion checks. `status` walks NOT_STARTED → IN_PROGRESS → COMPLETE.

**Report (student → school).** The `Report` is generated by reading the instance's
responses in section/block order — no re-entry. Student reviews, marks READY, and the
export (PDF/DOCX in Supabase storage) is uploaded to the school; status → UPLOADED_TO_SCHOOL,
with `schoolRef` holding the receipt/LMS id.

## Offline sync (the field-capture constraint)

Phillip Island has patchy coverage, so `Response` carries sync bookkeeping: `clientId`
(device-generated id), `updatedAt`, `syncedAt`, and a soft-delete `deleted` flag. The
device writes locally first and reconciles on reconnect. Pair Supabase with **PowerSync**
(or WatermelonDB if the client is React Native) — the exact library depends on the app
framework, which is the one open decision left.
